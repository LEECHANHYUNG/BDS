## 1. 개요 & 목표
- 만들려는 기능과 그로 인해 시스템이 수행하게 될 비즈니스 결과
- 이 계획이 달성하려는 것 / 달성하지 않는 것 (범위 경계)

## 2. 접근 방식
- 어떻게 구현할지에 대한 상세한 설명
- 기존 시스템(토큰/컴포넌트/패턴/캐싱·상태 레이어)과 어떻게 맞물리는지
- (레퍼런스 구현체가 제공되었다면) 그것을 어떻게 차용하는지

## 3. 변경될 파일 경로
실제 코드베이스에서 확인한 경로만 적는다. 확인하지 못한 경로는 `(가정)`으로 표시한다.

- `path/to/file.ts` — 무엇을 왜 바꾸는지
- `path/to/another.ts` (신규) — 무엇을 만드는지

## 4. 코드 스니펫
실제 변경될 코드를 보여준다. 기존 코드를 읽고 그 스타일·네이밍에 맞춰 쓴다. 발명한 예시가 아니라 실제 코드 기반이어야 한다.

```ts
// path/to/file.ts
```

## 5. 고려사항 & 트레이드오프
- 선택한 접근의 상충 관계, 대안과 그것을 택하지 않은 이유
- (해당 시) 도메인 고려사항 — 접근성, 규제, 숫자/통화 포맷, 신뢰성 등
- 위험 요소와 완화 방안
