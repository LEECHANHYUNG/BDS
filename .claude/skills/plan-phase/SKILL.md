# Plan Phase (계획 단계)

기능 구현은 **조사 → 계획 → 구현** 3단계로 진행된다. 이 스킬은 그 두 번째 단계인 **계획(Planning)** 을 실행한다. ([research-phase] 가 첫 단계다.)

계획 단계의 핵심은 단순히 plan.md를 한 번 쓰는 게 아니라, 그 문서를 **개발자와 Claude 사이의 '공유된 가변 상태(Shared Mutable State)'** 로 만드는 데 있다. Claude는 코드를 이해하고 솔루션을 제안하는 데 뛰어나지만, 개발자의 제품 우선순위·사용자 페인포인트·감수할 트레이드오프는 알지 못한다. 그래서 개발자가 문서에 직접 노트를 달아 판단력을 주입하는 **주석 사이클(Annotation Cycle)** 이 이 단계의 진짜 가치다.

**이 스킬은 계획 문서만 쓰고 수정한다. 절대 코드를 구현하지 않는다.** 가장 비싼 실패는 문법 오류가 아니라, "격리된 상태에서는 완벽히 작동하지만 기존 아키텍처를 망가뜨리는" 구현이다. 계획이 승인되기 전에 코드로 뛰어들면 정확히 그 실패가 발생한다.

## 워크플로우

### 1. 조사 산출물 연결 & 주제 확정

- 주제를 kebab-case 슬러그로 정한다 (예: `cursor-pagination`, `toast-system`).
- `docs/research/<슬러그>.md` 를 먼저 찾아 **읽고 계획의 토대로 삼는다.**
- 없으면 사용자에게 한 줄로 알리고 — _"조사 산출물(`docs/research/<슬러그>.md`)을 못 찾았습니다. 조사 없이 계획하면 정확도가 떨어질 수 있어요"_ — **그대로 진행할지 / 조사(research-phase)부터 할지** 선택하게 한다. 강제하지 않는다.
- 산출 경로: `docs/plans/<슬러그>.md`. `docs/plans/` 가 없으면 만든다. 같은 이름 파일이 이미 있으면 **덮어쓰지 말고** 사용자에게 알린 뒤 이어쓸지 새 슬러그를 쓸지 결정한다.

### 2. 실제 코드베이스 읽기 (작성 전 필수)

plan.md를 **쓰기 전에**, 수정 대상이 될 파일과 그 **호출부·이웃 코드**를 읽는다. 이건 선택이 아니다 — 구체적 코드를 읽지 않으면 Claude는 학습된 과거 지식으로 그럴싸하지만 기존 시스템과 어긋나는 계획을 세운다.

- 변경될 파일 경로와 코드 스니펫은 **실제로 확인한 코드**에 기반한다. 발명한 예시 금지.
- 확인하지 못한 경로·가정은 사실처럼 단정하지 말고 `(가정)`으로 명시한다.
- 사용자가 **레퍼런스 구현체**(오픈소스/내부 코드)를 함께 주면 읽고 차용한다. Claude는 맨땅 설계보다 구체적 레퍼런스가 있을 때 훨씬 잘 작동한다. (먼저 달라고 조르지는 않는다.)

### 3. plan.md 작성 — 고정 템플릿

`assets/plan-template.md` 의 골격을 그대로 따른다. 검토 표면이 매번 같은 구조여야 빠르게 검토된다. 섹션: 개요/목표 · 접근 방식 · 변경 파일 경로 · 코드 스니펫 · 고려사항/트레이드오프 · 할 일 목록.

**할 일 목록(6번 섹션)은 지금 확정하지 않는다.** 계획이 주석 사이클로 다듬어진 뒤, 승인 직전에 확정한다 (5단계 참고). 그전까지는 비워두거나 가벼운 초안만 둔다.

### 4. 핸드오프 — 정지하고 노트 요청

plan.md 저장 후 **멈춘다.** 다음과 같이 안내한다:

> _"`docs/plans/<슬러그>.md`에 계획을 작성했습니다. 에디터에서 문서를 열어 인라인 노트를 직접 달아주세요 — 잘못된 가정 수정, 접근 거부, 제약 추가, 제가 모르는 도메인 지식 주입 등 무엇이든요. 노트는 `<!-- NOTE: ... -->` 나 `> ` 인용으로 달면 확실하고, 자유롭게 본문 옆에 적어도 됩니다. 다 되면 알려주세요. **아직 구현하지 않습니다.**"_

매 핸드오프마다 **"아직 구현하지 않습니다"** 를 명시한다. 이 가드레일이 없으면 Claude는 계획이 "어느 정도 괜찮다"고 느끼는 순간 코딩으로 뛰어든다. 계획은 사용자가 명시적으로 승인하기 전까지 괜찮은 게 아니다.

### 5. 주석 사이클 — 노트 반영하고 다시 정지

사용자가 "노트 반영해줘" 라고 하면:

- 문서를 **전체 다시 읽어** 노트를 식별한다. `<!-- NOTE -->`·`>` 마커는 확실한 신호이고, 마커 없는 프리폼 노트(예: 옵션 옆에 _"옵션 아님"_ 두 단어)도 본문과 구분해 추론한다.
- **모든 노트를 반영한다. 마음에 드는 것만 골라 반영(체리픽)하지 않는다.** 각 노트는 가정 수정·접근 거부·제약 추가·방향 재지정 중 무엇인지 이해하고 그에 맞게 문서를 고친다.
- 반영 후 **소비한 노트는 제거**해 문서를 깨끗하게 유지하고, 노트별로 무엇을 어떻게 바꿨는지 **짧게 요약 보고**한다.
- 다시 **"아직 구현하지 않습니다"** 로 정지한다.

이 사이클을 **사용자가 명시적으로 승인할 때까지 반복**한다 (보통 1~6회). "승인", "이대로 가자", "좋아 확정", "구현 시작" 같은 명시 신호만 루프를 끝낸다. 어중간한 긍정("괜찮네")은 승인이 아니다 — 애매하면 확인한다.

### 6. 승인 시 — 할 일 목록 확정 & 구현 핸드오프 (실행 금지)

사용자가 명시 승인하면:

1. **할 일 목록을 확정한다.** 모든 단계를 완료하는 데 필요한 개별 작업을, 구현 중 진행 추적기로 쓸 수 있도록 아주 잘게 쪼개 `[ ]` 체크박스로 plan.md 6번 섹션에 적는다. 각 작업은 **검증 가능한 단위**(타입체크/테스트/실제 동작으로 확인되는 단위)로 쪼갠다 — 구현 단계의 자기검증 게이트가 작업 단위로 실행된다.
2. **"아직 구현하지 않았습니다"** 를 다시 확인한다. 이 스킬은 여기서 끝이며, 코드를 쓰지 않는다.
3. **구현은 Claude Code가 직접 하지 않는다.** 이 레포의 역할 분담상 대량 컴포넌트 구현·빌드·시각 산출은 **Codex에 핸드오프**한다 (`CLAUDE.md`·`docs/collab-protocol.md` 참고). 다음 단계는 `implement-phase` 스킬이 맡는다 — 승인된 계획을 `PROJECT.md` 현황판에 올리고 Codex에 넘긴 뒤, Claude Code는 설계 의도 준수와 리스크를 리뷰한다. 사용자에게 _"구현 핸드오프로 넘어가려면 `implement-phase`를 실행하세요"_ 라고 안내한다.

   Codex(또는 별도 실행 주체)에 줄 **표준 구현 명령 프롬프트**를 복붙용으로 함께 제시한다 (이 스킬은 실행하지 않는다):

> _"`docs/plans/<슬러그>.md` 계획을 모두 구현해라. 한 작업이 끝날 때마다 해당 작업을 타입체크·테스트·(해당 시) 실제 동작으로 검증한 뒤 계획서에 완료 표시(`[x]`)를 하고, 검증이 통과하면 논리 단위로 커밋해라. 모든 작업과 단계가 완료될 때까지 중간에 확인받으려고 멈추지 마라. 불필요한 주석이나 JSDoc은 달지 말고, `any`나 `unknown` 타입은 쓰지 마라. 새로운 문제가 생기지 않도록 계속 타입 체크(typecheck)를 실행하면서 진행해라."_

## 왜 계획 모드가 아니라 .md 파일인가

Claude Code 내장 계획 모드 대신 프로젝트 내 실제 `.md` 파일을 쓰는 이유: 완전한 통제권. 에디터에서 직접 편집하고, 인라인 노트를 달고, 프로젝트에 영구 유산(Artifact)으로 남길 수 있다. 대화창으로 방향을 조종하면 과거 결정을 보려고 한참 스크롤해야 하지만, 계획서는 거시적으로 한눈에 검토할 수 있는 구조화된 명세다.

## 스스로 점검할 규칙

- [ ] **작성 전 실제 코드 읽기.** 수정 대상과 그 주변을 읽고, 경로·스니펫은 실제 코드 기반. 미확인은 `(가정)`.
- [ ] **고정 템플릿 준수.** 검토 표면이 매번 같은 구조.
- [ ] **할 일 목록은 승인 직전 마지막에 확정.**
- [ ] **모든 노트 반영, 체리픽 금지.** 반영 후 노트 제거 + 변경 요약.
- [ ] **매 핸드오프마다 "아직 구현하지 않습니다" 명시.**
- [ ] **명시 승인으로만 루프 종료.** 어중간한 긍정은 승인 아님.
- [ ] **절대 코드 구현 금지.** 승인 후에도 plan.md만 확정하고 구현 프롬프트를 넘긴다.
