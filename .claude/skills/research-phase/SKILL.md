# Research Phase (조사 및 탐색)

기능 구현은 계획 → 구현이 아니라 **조사 → 계획 → 구현** 3단계로 진행된다. 이 스킬은 그 첫 단계인 **조사(Research & Exploration)** 를 실행한다. 조사가 틀리면 계획이 틀어지고, 계획이 틀어지면 구현이 망가진다(Garbage in, garbage out). 그래서 이 단계의 산출물 `docs/research/<주제>.md` 는 **인간 개발자의 '검토 표면(Review Surface)'** 이 된다.

이 스킬은 **조사하고 기록까지만** 한다. **코드는 절대 쓰지 않는다.**

## 무엇을 조사할지: 의도에서 범위를 추론한다

조사는 두 축으로 나뉘며, **서로 독립적**이다. 사용자 요청 문구에서 어느 축이 필요한지 추론한다.

| 요청 성격 | 코드베이스 축 | 웹 축 |
|---|---|---|
| "이 기능을 구현하려고 조사해줘" (구현 목적) | ✅ | ✅ |
| "이 모듈/컴포넌트/시스템을 깊이 이해해줘" | ✅ | ❌ |
| "이 라이브러리/기술을 도입하려는데 최신 모범 사례 조사" | ❌ (또는 가볍게) | ✅ |
| "이 버그/이슈와 유사한 오픈소스 해결책 찾아줘" | ✅ (관련 흐름) | ✅ |

애매하면 **둘 다** 수행한다. 어느 축을 수행할지 시작할 때 한 줄로 사용자에게 알린다 (예: "코드베이스만 조사합니다" / "웹 조사만 합니다" / "둘 다 조사합니다").

## 워크플로우

### 1. 주제 슬러그와 파일 경로 확정
- 조사 주제를 kebab-case 슬러그로 정한다 (예: `button-tokens`, `cursor-pagination`, `toast-system`).
- 산출 경로: `docs/research/<슬러그>.md`. `docs/research/` 가 없으면 만든다.
- 같은 이름 파일이 이미 있으면 **덮어쓰지 말고** 사용자에게 알린 뒤 이어쓸지 새 슬러그를 쓸지 결정한다.

### 2. 코드베이스 축 (해당할 때)
관련 디렉터리/파일을 **겉핥기 아닌 심층으로** 읽는다. 기존 토큰·컴포넌트·패턴·네이밍 컨벤션·캐싱/상태 관리 레이어를 파악하고, 새 기능이 **기존 시스템과 어떻게 맞물리는지** 이해한다. 발견한 모든 관련 파일은 `path:line` 형식으로 기록한다.

### 3. 웹 축 (해당할 때) — 서브에이전트에 위임
**메인 에이전트가 직접 WebSearch 하지 않는다.** `Agent` 툴로 웹 검색 서브에이전트를 띄워 위임한다. 조사 주제가 여러 개면 **여러 서브에이전트를 병렬로** 띄운다. 서브에이전트에게 보낼 프롬프트에는 아래 [조사 강제 규칙](#조사-강제-규칙)을 반드시 포함한다. 서브에이전트는 정리된 발견 + 출처 URL 목록을 반환하게 한다.

### 4. research.md 작성
아래 [문서 구조](#researchmd-문서-구조)대로 종합해 기록한다.

### 5. 종료 — 검토 요청 후 정지
- `docs/research/<슬러그>.md` 저장 완료를 알린다.
- 핵심 발견을 **3~6줄로 요약**해 대화창에 보여준다.
- 다음과 같이 안내하고 **멈춘다**: *"조사 결과를 `docs/research/<슬러그>.md`에 기록했습니다. 검토해 보시고 잘못된 이해나 방향이 있으면 알려주세요. 검증이 끝나면 계획(plan) 단계로 넘어가면 됩니다."*
- **여기서 계획을 세우거나 코드를 쓰지 않는다.** 사용자가 명시적으로 다음 단계를 요청할 때까지 정지.

## 조사 강제 규칙

원문의 핵심 통찰: 구체적 제약을 주지 않으면 AI는 코드를 겉핥기로 읽거나 학습된 과거 지식(정적 지식)에만 의존해 버린다. 아래 규칙은 코드베이스 읽기와 서브에이전트 프롬프트 **양쪽 모두**에 적용한다.

- [ ] **표면적 분석 금지.** 관련 코드는 깊이(deeply) 읽고, 주변 시스템과의 연결까지 이해한다. 핵심 파일은 일부만 보고 넘기지 않는다.
- [ ] **정적 지식 의존 금지.** 외부 기술은 학습된 기억이 아니라 **실제 웹 검색으로** 확인한다.
- [ ] **최신 버전·deprecation 확인 필수.** 인터넷에서 본 예전 방식을 그대로 적용하지 않는다. 현재 권장 방식인지, deprecated 되지 않았는지 검증한다.
- [ ] **출처 URL 명시.** 외부 조사 항목마다 출처 링크를 남긴다. 근거 없는 주장 금지.
- [ ] **조사 완료 전 정지 금지.** 조사 도중 멈추지 말고, 충분히 깊이 파악할 때까지 계속한다.
- [ ] **구현 금지.** 이 단계에서는 코드를 쓰지 않는다.

## research.md 문서 구조

다음 5개 섹션 골격을 항상 따른다. 검토 표면이 매번 같은 구조여야 빠르게 검토된다. 해당 없는 축의 섹션(2 또는 3)은 비운다.

```markdown
# Research: <주제>

> 작성: <스킬 실행 시점> · 상태: 검토 대기

## 1. 조사 범위 & 질문
- 무엇을, 왜 조사했는가
- 답하려는 핵심 질문 목록

## 2. 코드베이스 분석
- 관련 기존 토큰/컴포넌트/패턴 (path:line 포함)
- 새 기능이 기존 시스템과 맞물리는 지점, 주의해야 할 레이어(캐싱/상태/네이밍 등)

## 3. 외부 조사 — 최신 동향 & 모범 사례
- 발견 항목마다 출처 URL
- 라이브러리 버전, deprecated 여부, 현재 권장 방식

## 4. 종합 발견 & 권장 방향
- 위 둘을 종합한 결론과 추천 접근
- (해당 시) 금융 도메인 고려사항 — 접근성, 규제, 숫자/통화 포맷, 신뢰성 등을 여기에 녹인다
- 트레이드오프

## 5. 미해결 질문 / 개발자 검토 필요 지점
- AI가 확신하지 못한 부분, 인간 판단이 필요한 결정
```

## 다음 단계 (참고)

조사가 검증되면 다음 단계로 넘어간다. 이 스킬 범위 밖이다.

- 다음: 계획 단계 — `plan-phase` 스킬이 이 조사 산출물을 토대로 `docs/plans/<주제>.md` 를 만들고 주석 사이클로 다듬는다.
- 그다음: 구현 단계 — `implement-phase` 스킬이 승인된 계획을 Codex에 핸드오프한다. (이 레포는 조사·계획=Claude, 구현=Codex 역할 분담)
