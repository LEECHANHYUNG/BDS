# Implement Phase (구현 단계 — Codex 핸드오프 & 리뷰)

기능 구현은 **조사 → 계획 → 구현** 3단계로 진행된다. 이 스킬은 그 마지막 단계인 **구현(Implementation)** 을 다룬다. ([research-phase] 가 1단계, [plan-phase] 가 2단계다.)

**이 레포의 역할 분담상 Claude Code는 코드를 직접 짜지 않는다.** 대량 컴포넌트 구현·빌드·시각 산출은 **Codex(실행)** 가 맡고, Claude Code(판단)는 **승인된 계획을 Codex에 핸드오프하고, 그 결과를 리뷰**한다. (근거: [`CLAUDE.md`]·[`AGENTS.md`]·[`docs/collab-protocol.md`].)

이 단계의 철학은 단 하나다: **구현은 창의적인 작업이 아니라 기계적인 작업이어야 한다.** 창의적인 결정은 조사와 주석 사이클에서 전부 내려졌다. 계획이 올바르면 실행은 지루하고 단순하다. "구현 중에 새 설계 결정이 필요하다"면 계획이 덜 된 것이다 — 코드로 메우지 말고 [plan-phase] 로 되돌린다.

## 전제 조건 — 승인된 계획서

이 스킬은 **승인된 `docs/plans/<슬러그>.md` 가 있을 때만** 실행한다.

- 계획서를 먼저 찾아 **전체를 읽는다.** 6번 할 일 목록이 확정돼 있는지 확인한다.
- 계획서가 없으면: *"핸드오프하려면 승인된 계획서(`docs/plans/<슬러그>.md`)가 필요합니다. `plan-phase`로 계획부터 세울까요?"* 라고 안내하고 **멈춘다.**
- 할 일 목록이 비었거나 주석(`<!-- NOTE -->`·`>`)이 남아 있으면 아직 주석 사이클이 안 끝난 것 — [plan-phase] 로 되돌린다.
- 승인이 애매하면(어중간한 긍정) 핸드오프하지 말고 확인한다. "구현해라", "이대로 가자", "Codex에 넘겨" 같은 **명시 신호**만 이 스킬을 연다.

## 워크플로우

### 1. 현황판 갱신 & 파일 락
[`PROJECT.md`] 를 연다.

- **Goal** — 이번 작업 목표를 한 줄로 적거나 갱신한다.
- **Work Board** — 구현 대상 파일(보통 `packages/**`)을 `Codex · 작업 중 ✓ locked` 로 등록한다. 이때부터 **Claude Code는 그 파일을 직접 수정하지 않고 리뷰만** 한다.
- 계획서(`docs/plans/<슬러그>.md`)는 Claude Code가 잡은 락이므로 Codex는 참고만 한다 — 핸드오프 프롬프트에 그 사실을 명시한다.

### 2. Codex에 핸드오프 — 표준 구현 명령
Codex에 줄 **구현 명령 프롬프트**를 제시한다. 이 한 줄에 원문의 모든 규칙이 담겨 있다:

> *"`docs/plans/<슬러그>.md` 계획을 모두 구현해라. 한 작업이 끝날 때마다 그 작업을 타입체크·테스트·(해당 시) 실제 동작으로 검증한 뒤 계획서에 완료 표시(`[x]`)를 하고, 검증이 통과하면 논리 단위로 커밋해라. 모든 작업과 단계가 완료될 때까지 중간에 확인받으려고 멈추지 마라. 계획에 없는 API·토큰·구조를 임의로 발명하지 말고, 모호하면 핸드오프로 되물어라. 불필요한 주석·JSDoc은 달지 말고, `any`/`unknown` 타입은 쓰지 마라. 계속 타입 체크를 실행하고, `pnpm check`로 정리해라. 긴 출력 명령은 `rtk`를 붙여 실행해라."*

각 구절의 의도:
- **"모두 구현해라"** — 계획에 있는 모든 것을 한다. 임의 취사선택 금지.
- **"검증 후 완료 표시"** — 계획서가 진행의 **유일한 진실의 원천(Source of truth)**. 검증 통과한 작업만 `[x]`.
- **"멈추지 마라"** — 계획 안에서는 중간 확인으로 흐름을 끊지 않는다. (단 [멈춰야 할 때](#멈춰야-할-때) 는 예외.)
- **"발명 금지"** — 계획에 없는 설계는 만들지 않는다. ([`AGENTS.md`] 의 핵심 규칙.)
- **"검증·정리·rtk"** — 이 레포 규약: 타입체크/테스트 + Biome(`pnpm check`) + RTK 토큰 절감.

스킬을 쓰지 않고 사용자가 직접 Codex 세션에 붙여넣을 수 있도록 이 프롬프트를 **복붙용으로** 내어준다.

### 3. Codex 구현물 리뷰 (Claude Code의 본분)
Codex가 구현·핸드오프하면, Claude Code는 **락을 잡지 않은 채 읽기·리뷰만** 하고 다음을 점검한다 ([`CLAUDE.md`] 리스크 검토 기준):

- [ ] **계획 준수** — 승인된 `docs/plans/<슬러그>.md` 의 접근·API·할 일 목록을 그대로 따랐는가. 발명한 API·토큰·구조는 없는가.
- [ ] **접근성** — 대비비, 키보드 내비게이션, ARIA, 포커스 관리가 컴포넌트 API에 녹아 있는가.
- [ ] **데이터 표시** — 숫자·날짜 포맷 일관성, 신뢰를 해치는 모호한 상태 표시, 정책·표현 위험은 없는가.
- [ ] **파급 영향** — 토큰/컴포넌트 변경이 의존 컴포넌트·앱에 미치는 영향. breaking change면 Changeset(`pnpm changeset`)이 있는가.
- [ ] **검증 실재** — 빌드·타입·테스트가 실제로 통과했는가("미검증"이면 통과를 요구). 시각 산출물은 [`DESIGN.md`] 기준·과장 표현 금지를 지켰는가.

리뷰 지적은 구현 단계답게 **짧고 거칠게** 한 문장으로 핸드오프한다 (아래 [리뷰 피드백](#리뷰-피드백)).

### 4. Handoff Log 갱신 & 락 해제
[`PROJECT.md`] 의 `Handoff Log` 에 한 줄 추가하고(최신이 위), 완료된 파일의 락을 해제한다. 형식:

```
[담당 도구] changed: <수정 파일> / verified: <검증 상태> / next: <다음 담당과 할 일>
```

예: `[Claude Code] changed: docs/plans/toast-system.md / verified: 계획 승인됨 / next: Codex가 packages/ui에 Toast 구현`

모든 작업이 `[x]` 가 되고 리뷰가 통과하면, `next` 를 비워(작업 종료) 마무리하고 무엇을 구현·검증·리뷰했는지 간결히 보고한다.

## 리뷰 피드백 — 역할이 아키텍트에서 감독관으로

구현이 시작되면 Claude Code의 역할은 설계자에서 감독관으로 바뀌고, 핸드오프는 짧아진다.

- *"`deduplicateByTitle` 함수가 계획에 있는데 구현 안 됐어. 추가해줘."* — 누락 보완
- *"설정 페이지가 메인 앱에 있는데 어드민 앱에 있어야 해. 옮겨줘."* — 위치 교정
- *"이 테이블은 기존 `users` 테이블과 헤더·페이지네이션·행 밀도까지 똑같이."* — 기존 코드 레퍼런스 (리뷰 시 그 파일을 먼저 읽고 비교한다)
- (UI) *"더 넓게", "여전히 잘림", "2px 틈 있음"* — 시각 문제는 스크린샷이 말보다 빠르다.

### 잘못된 방향이면 — 깁지 말고 되돌린다
구현이 완전히 틀어졌을 때 코드를 기워서 고치려 하지 않는다. Codex에 **git 변경을 폐기(discard)해 마지막 정상 커밋으로 되돌린 뒤 범위를 좁혀 다시** 지시한다:

> *"방금 한 거 다 되돌리고, 리스트 뷰를 미니멀하게 만드는 것에만 집중해 — 다른 건 하지 마."*

논리 단위 커밋(게임의 세이브 포인트)이 이 롤백을 값싸게 만든다. 점진적으로 깁는 것보다 거의 항상 낫다.

## 멈춰야 할 때

"멈추지 마라"는 **계획 안에서**의 기계적 실행에 적용된다. 다음은 예외 — 핸드오프를 멈추고 계획으로 되돌린다:

- **계획에 없는 설계 결정이 필요할 때.** 임의로 정하지 말고 [plan-phase] 로.
- **계획이 실제 코드와 어긋남을 발견했을 때.** 추측으로 메우지 말고 보고 후 계획 갱신 여부를 묻는다.
- **검증이 계속 실패하고 원인이 계획의 결함일 때.** 우회로 덮지 않는다.
- **범위를 벗어나는 변경이 필요해 보일 때.** 사용자가 범위를 조정하게 둔다.

## 스스로 점검할 규칙

- [ ] **승인된 계획서가 있을 때만 시작.** 없거나 주석이 남았으면 plan-phase로 되돌린다.
- [ ] **Claude Code는 구현하지 않는다.** Codex에 핸드오프하고 리뷰만 한다. Codex가 락 잡은 `packages/**` 는 직접 수정 금지.
- [ ] **PROJECT.md 현황판 갱신.** 락 잡기 → 핸드오프 → Handoff Log 기록 → 락 해제.
- [ ] **계획대로만 구현 지시.** 새 설계 결정은 멈추고 계획으로.
- [ ] **검증 실재 확인.** "미검증" 핸드오프는 통과를 요구한다.
- [ ] **리뷰 5축**(계획 준수·접근성·데이터 표시·파급 영향·검증)으로 점검.
- [ ] **잘못된 방향은 깁지 말고 되돌리고 범위를 좁힌다.**
