# 포트폴리오용 공개 DS — 능력 우선순위 C>A>B와 dogfooding 데모 앱

이 디자인 시스템은 실제 외부 소비자를 위한 것이 아니라 **포트폴리오/오픈소스 명함**으로서 공개 npm publish 자체가 산출물이다. 따라서 증명할 능력의 우선순위를 **C(금융 도메인) > A(DS 아키텍처) > B(FE 인프라)** 로 고정한다 — 우선순위 없는 "다 잘함"은 포폴에서 "특징 없음"으로 읽히고, 가장 쉬운 인프라(B)에 시간이 쏠리는 함정을 막기 위함이다.

소비자가 0개인 채로 공개 라이브러리를 만들면 API를 검증할 길이 없으므로, 모노레포에 **얇은 데모 앱(`apps/`)을 두어 그 앱이 첫 소비자(dogfooding)** 가 되게 한다. 데모 앱이 라이브러리(`packages/`)를 실제로 조합·검증하는 동시에, "금융 화면을 안다"는 것을 보여주는 포폴 산출물이 된다. 이 dogfooding 필요성이 모노레포 구조를 정당화한다(인프라를 위한 인프라가 아님).

## Considered Options

- **dogfooding 없이 Storybook/문서만**: 빠르지만 컴포넌트 조합에서 터지는 설계 결함을 못 봄. C 증명이 약해 기각.
- **내 실제 앱을 먼저 만들고 점진 공개**: 가장 견고하나 포폴 목적(공개 자체가 산출물)에 비해 과함. 얇은 데모로 절충.
