# Plan: 디자인 시스템 PR 템플릿

> 작성: 2026-06-14 · 상태: 완료
> 근거 조사: docs/research/pr-template.md

## 1. 개요 & 목표

현재 BDS 레포에는 GitHub PR 템플릿이 없다(`.github/` 디렉터리 부재). 모든 PR이 일관된 검토 표면을 갖도록 **단일 PR 템플릿**을 도입한다.

**달성하려는 것 (범위 안)**

- `.github/PULL_REQUEST_TEMPLATE.md` 단일 파일 생성. PR 생성 시 자동으로 본문에 채워진다.
- 변경 유형·요약·시각 변경은 펼친 형태로, 디자인 시스템·금융 도메인 리스크 체크리스트는 `<details>`로 접어 "길어서 안 채워지는" 안티패턴을 회피한다.
- 기존 `docs/templates.md`의 리스크 노트 4축(접근성/금융 도메인/파급 영향/과장·추정)과 정렬하고, `DESIGN.md`·Changesets 흐름을 PR 게이트로 반영한다.
- 한국어로 작성(레포 규칙).

**달성하지 않는 것 (범위 밖 — 추후 별도 결정)**

- 이슈 템플릿(YAML forms)은 만들지 않는다.
- changeset-bot(GitHub App) 설치·CI hard block은 도입하지 않는다. PR 템플릿의 자가 점검 체크박스로만 안내한다. (packages가 아직 비어있어 봇 효용이 낮음 — 컴포넌트 패키지 도입 시 재검토.)
- AI 협업(Claude/Codex Handoff Log) 항목은 템플릿에 넣지 않는다. 협업 추적은 PROJECT.md/collab-protocol.md가 계속 담당.
- 다중 템플릿(`.github/PULL_REQUEST_TEMPLATE/` + `?template=`)은 쓰지 않는다. 변경 유형은 단일 템플릿 내 체크박스로 구분.

## 2. 접근 방식

### 2.1 저장 위치 & 메커니즘

- 경로: `.github/PULL_REQUEST_TEMPLATE.md` (표준 숨김 디렉터리). 단일 템플릿이므로 PR 생성 시 본문에 자동 로드된다.
- 템플릿은 **기본 브랜치(main)에 머지돼야** 노출된다(조사 §3.1). 이 PR 자체에는 적용되지 않고, 머지 이후 PR부터 적용된다.
- YAML 폼은 PR에 미지원(조사 §3.1)이라 Markdown으로 작성한다.

### 2.2 구조 — "강제 입력은 적게, 자가 점검 체크리스트는 접어서"

조사 §4.5 절충안을 따른다. 펼친 핵심 섹션 + 접은 체크리스트:

- **펼침(항상 보임)**: `Closes #` / 변경 유형 / 무엇을·왜 / 시각적 변경(표).
- **접음(`<details>`)**: 체크리스트 — 일상적으로 훑되 본문 노이즈를 줄인다.

### 2.3 체크리스트를 레포 자산과 정렬

조사 §2.4에서 확인한 기존 자산에 1:1로 연결한다:

- **접근성** ← `DESIGN.md` "접근성 기본 탑재" + `docs/templates.md` 리스크 노트 "접근성 위험". 대비비·키보드·ARIA·포커스.
- **금융 도메인** ← `DESIGN.md` "신뢰 우선/명료함" + 리스크 노트 "금융 도메인 위험". 숫자/통화/날짜 포맷, 모호한 상태 표시 회귀 없음.
- **의미 토큰 강제** ← `DESIGN.md` "일관성: 토큰 우회 하드코딩 금지". raw 색상값·임의 픽셀 금지.
- **파급 영향/changeset** ← 리스크 노트 "파급 영향" + `.changeset/config.json`·`package.json:scripts`(`pnpm changeset`). breaking 영향 + changeset 추가.
- **테스트/문서** ← `package.json`(`turbo run lint`) + 향후 Storybook(현재 미도입이라 "해당 시"로 조건부).

### 2.4 업계 패턴 차용 (조사 §3.2)

- before/after는 표 형태(Polaris·Primer). GIF는 접근성 위해 `<details>`로 감싸라는 안내 주석(Polaris).
- breaking change는 변경 유형 체크 + 본문에 "마이그레이션 경로" 안내(Chakra). semver 4단계 Rollout(Primer)까지는 가지 않는다 — 현재 packages가 비어있어 과하고, 중간 톤 결정과 어긋남.
- 머리말 주석으로 "PR은 한 유형으로 작게"(Chakra) 안내.

## 3. 변경될 파일 경로

실제 코드베이스에서 확인한 경로만 적는다.

- `.github/PULL_REQUEST_TEMPLATE.md` (신규) — 단일 PR 템플릿 본문. `.github/` 디렉터리도 이번에 새로 생긴다(현재 부재, 루트 `ls`로 확인).

그 외 파일은 변경하지 않는다. (`docs/templates.md`에 "PR 템플릿" 참조 한 줄을 추가할지는 §5 미해결 질문으로 남긴다 — 기본은 PR 템플릿 파일만.)

## 4. 코드 스니펫

실제 도입할 `.github/PULL_REQUEST_TEMPLATE.md` 전문 초안. 길이=중간, 체크리스트는 접음, 한국어.

```markdown
<!--
PR은 한 가지 유형으로 작게 유지해 주세요. 큰 변경은 나눠서 올리면 리뷰가 빨라집니다.
UI 변경이라면 아래 "시각적 변경"에 before/after를 첨부해 주세요.
-->

Closes #

## 변경 유형

- [ ] feat — 신규 기능/컴포넌트/토큰
- [ ] fix — 버그 수정
- [ ] docs / chore — 문서·설정 등
- [ ] breaking change — 다운스트림에 영향 (아래에 마이그레이션 경로를 적어 주세요)
```

## 5. 고려사항 & 트레이드오프

- **길이 절충**: 미니멀(MUI/Radix)은 채우기 쉽지만 디자인 시스템 고유 리스크(접근성·breaking·금융 포맷)를 놓치기 쉽고, 두꺼운 형(Primer/Carbon)은 무시되기 쉽다. → 핵심은 펼치고 체크리스트는 접는 중간안 채택(사용자 승인 완료).
- **금융 도메인**: 일반 디자인 시스템 템플릿엔 없는 "숫자/통화/날짜 포맷 회귀" "의미 토큰 강제" 항목을 명시 → `DESIGN.md` 신뢰·명료·일관성 원칙을 PR 게이트로 끌어올림. 이게 BDS 템플릿의 차별점.
- **접근성**: 거의 모든 디자인 시스템이 두는 항목. 금융 제품 신뢰성과 직결되므로 체크리스트 상단부에 명시.
- **changeset 강제 수준(위험)**: 템플릿 체크박스는 자가 점검일 뿐 강제력이 없다. `commit:false`라 누락 시 릴리스에서 빠진다(조사 §2.2). 완화: 컴포넌트 패키지 도입 시점에 changeset-bot(비차단 알림) 도입을 재검토. 지금은 packages가 비어있어 과한 설정 회피.
- **Storybook/테스트 조건부**: 현재 둘 다 미도입 상태라 "해당 시"로 표기. 도입되면 "해당 시"를 떼는 후속 갱신이 필요(소소한 유지보수 부채).
- **적용 시점 함정**: 템플릿은 main 머지 후의 PR부터 적용된다. 이 PR 자체에는 안 보이는 게 정상.
