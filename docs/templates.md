# 산출물 템플릿

협업 산출물은 매번 같은 구조를 쓴다. 검토 표면이 일정해야 사람이 빠르게 검토하고, 두 도구가 서로의 산출물을 예측할 수 있다. 아래 템플릿을 복사해 채운다.

- 조사(`docs/research/*.md`)·계획(`docs/plans/*.md`) 문서 구조는 각각 `research-phase`·`plan-phase` 스킬과 `plan-phase/assets/plan-template.md`가 이미 정의한다. 여기서는 그 외 협업 산출물(컴포넌트 설계서·리스크 노트·핸드오프)을 다룬다.

---

## 컴포넌트 설계서 (Claude Code 작성 → Codex 구현)

계획 단계에서 컴포넌트 하나의 공개 API·동작을 확정하는 문서. Codex는 이 설계서를 발명 없이 그대로 구현한다.

```markdown
# 컴포넌트 설계: <컴포넌트명>

> 상태: 검토 대기 | 승인됨 · 계획: docs/plans/<주제>.md

## 목적 / 사용 맥락
- 어떤 화면에서, 어떤 문제를 푸는 컴포넌트인가

## 공개 API (Props)
| prop | 타입 | 기본값 | 설명 |
| --- | --- | --- | --- |

## 상태 / 변형 (Variants)
- 시각·동작 변형 목록 (예: size, tone, disabled, loading)

## 접근성 요구사항
- 키보드 동작, ARIA 역할/속성, 포커스 관리, 색 대비

## 사용 토큰
- 어떤 의미 토큰을 쓰는가 (DESIGN.md 기준, raw 값 금지)

## 엣지 케이스 / 비목표
- 처리해야 할 경계, 이번에 다루지 않는 것
```

## 리스크 노트 (Claude Code 작성)

구현·설계의 위험을 점검한 문서. 데이터·근거 없는 단정 대신 위험과 미해결 질문을 명시한다.

```markdown
# 리스크 노트: <주제>

> 대상: <리뷰한 산출물 경로> · 작성: <시점>

## 접근성 위험
- 대비/키보드/ARIA에서 놓친 점

## 제품 맥락 위험
- 숫자·날짜 포맷, 모호한 상태 표시, 정책·표현 위험

## 파급 영향
- 변경이 의존 컴포넌트·앱에 미치는 영향, breaking 여부, Changeset 필요성

## 과장·추정 영역
- 데이터 없이 단정한 부분, 검증이 필요한 가정

## 권장 조치
- 위험별 대응(반영 / 보류 / 사람 판단 필요)
```

## 핸드오프 한 줄 (PROJECT.md Handoff Log)

작업을 넘길 때 `PROJECT.md`의 `Handoff Log`에 추가한다.

```
[담당 도구] changed: <수정 파일> / verified: <검증 상태> / next: <다음 담당과 할 일>
```

- `verified`: 무엇으로 검증했는지(빌드 통과 / 테스트 통과 / 계획 승인 / 미검증).
- `next`: 다음 담당과 할 일. 비면 작업 종료.
